### C. Tax/GST Endpoints

- **Base Route URL**: `/api` (registered in `index.js`)
- **Endpoints Defined**:
  - `GET /api/taxes`: Get all non-deleted taxes for the user. Returns `{ taxes: [...] }`.
  - `GET /api/taxes/:id`: Retrieve details of a specific tax rule.
  - `POST /api/taxes`: Create a new tax rate. Required body parameters: `tax_name` and `rate`. Optional parameters: `tax_type`, `description`, `status`.
  - `PUT /api/taxes/:id`: Update fields on a tax.
  - `DELETE /api/taxes/:id`: Soft delete a tax rule (`is_deleted` is updated to `true`).
- **Tax Model Schema**:
  - `id` (SERIAL PRIMARY KEY)
  - `user_id` (INTEGER NOT NULL)
  - `tax_name` (VARCHAR NOT NULL)
  - `tax_type` (VARCHAR: e.g., GST, CGST, SGST, IGST, Other)
  - `rate` (NUMERIC(5,2) NOT NULL DEFAULT 0)
  - `description` (TEXT)
  - `status` (VARCHAR DEFAULT 'active')
  - `is_deleted` (BOOLEAN DEFAULT false)
  - `created_at` / `updated_at` (TIMESTAMP)

## 3. Database Table Discrepancy & Resolution

### The Discrepancy
There is a clear mismatch in the backend code between the SQL DDL statements and the SQL INSERT statements regarding the `inventory_movements` table columns:

1. **Table Creation (DDL)** in `src/controllers/inventoryController.js` (line 15-28):
   ```sql
   CREATE TABLE IF NOT EXISTS inventory_movements (
     id SERIAL PRIMARY KEY,
     user_id INTEGER NOT NULL,
     item_id INTEGER NOT NULL,
     movement_type VARCHAR(30) NOT NULL,
     quantity NUMERIC(12,2) NOT NULL,
     previous_stock NUMERIC(12,2) DEFAULT 0,
     new_stock NUMERIC(12,2) DEFAULT 0,
     reason VARCHAR(255),
     reference_number VARCHAR(100),
     notes TEXT,
     created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
   )
   ```
   *Uses: `movement_type`, `quantity`, `reason`, `notes`, `created_at`*

2. **Data Insertion (SQL INSERTS)** across backend controllers:
   - `inventoryController.js` (`createMovement` at line 123):
     ```sql
     INSERT INTO inventory_movements (user_id, item_id, transaction_type, quantity_change, reference_number, entry_date, description) ...
     ```
   - `deliveryChallanController.js` (at line 400):
     ```sql
     INSERT INTO inventory_movements (user_id, item_id, transaction_type, quantity_change, reference_number, entry_date, description) ...
     ```
   - `itemController.js` (at line 116):
     ```sql
     INSERT INTO inventory_movements (user_id, item_id, transaction_type, quantity_change, entry_date, description) ...
     ```
   *Uses: `transaction_type`, `quantity_change`, `entry_date`, `description`*

### Resolution
- **Actual DB State**: Because all INSERT statements inside the controllers use `transaction_type` and `quantity_change`, the database table columns in production must be using `transaction_type` and `quantity_change` (along with `entry_date` and `description`) for the write operations to function successfully.
- **Client-Side Model Handling**: The Flutter client contains specific fallback mapping logic in `lib/features/inventory/domain/inventory_movement_model.dart`:
  ```dart
  factory InventoryMovement.fromJson(Map<String, dynamic> json) {
    final type = json['transaction_type'] as String? ?? json['movement_type'] as String? ?? 'adjustment';
    final qty = (json['quantity_change'] as num? ?? json['quantity'] as num? ?? 0).toDouble();
    final desc = json['description'] as String? ?? json['reason'] as String?;
    final entryDate = json['entry_date'] as String? ?? json['created_at'] as String? ?? '';
    ...
  }
  ```
  This implementation ensures that the client remains robust and functions seamlessly regardless of which naming variant is present in the database table schema.
